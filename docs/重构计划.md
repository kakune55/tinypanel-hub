# tinypanel-hub 重构计划

目标：保持项目轻量、快速、易维护，同时为后续前端 UI、多用户账号和设备绑定留出清晰边界。

## 当前模型

核心关系：

```text
User -> Device -> Message
```

- `User` 是账号身份，也是当前阶段的租户边界。
- `Device` 是用户绑定的硬件设备。
- `Message` 是投递给某个设备的收件箱消息。
- 不使用数据库，继续使用 `state.json + telemetry JSONL`。
- 不再使用频道或订阅模型。

## API 分组

用户侧 API：

```http
GET    /api/v1/me
GET    /api/v1/devices
POST   /api/v1/devices/bind
GET    /api/v1/devices/{device_id}
PATCH  /api/v1/devices/{device_id}
DELETE /api/v1/devices/{device_id}
POST   /api/v1/devices/{device_id}/messages
GET    /api/v1/devices/{device_id}/messages
GET    /api/v1/devices/{device_id}/telemetry
```

设备侧 API：

```http
POST /api/v1/device/hello
GET  /api/v1/device/messages
POST /api/v1/device/messages/ack
POST /api/v1/device/telemetry
POST /api/v1/device/telemetry/batch
GET  /api/v1/device/snapshot
```

管理侧 API：

```http
POST /api/v1/admin/users
```

## 认证

- 用户 API 使用 `Authorization: Bearer <user-token>`。
- 设备 API 使用 `X-Device-ID` 和 `X-Device-Secret`。
- 管理 API 使用配置里的 `server.api_token`。
- token 和 device secret 在状态文件中只保存 hash。

## 存储

- `state.json` 保存 users、devices、messages、todos、weather 和自增 ID。
- telemetry 继续写入 JSONL。
- 查询时按 `owner_id` 和 `device_id` 做隔离。
- 后续如文件存储变复杂，再考虑 bbolt 或纯 Go SQLite。

## 后续前端方向

- `web/` 放前端源码。
- `internal/webui/dist/` 放生产构建产物。
- Go 服务托管前端页面，API 固定在 `/api/v1/*`。
