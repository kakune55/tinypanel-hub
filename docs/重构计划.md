# tinypanel-hub 重构计划

目标：保持项目轻量、快速、易维护，同时为后续前端 UI 和更多设备协议扩展留出清晰边界。

## 原则

- 优先小步重构，不一次性推翻现有代码。
- 保留当前 REST + JSON 协议的兼容性。
- 使用 `go-chi/chi/v5` 只强化路由和中间件组织，不引入重型框架。
- 存储继续以文件为主，后续根据数据量再考虑替换。
- 前端 UI 作为独立构建产物，最终由 Go 服务嵌入和托管。

## 阶段一：chi 路由骨架

- 引入 `github.com/go-chi/chi/v5`。
- 将当前 `http.ServeMux` 路由迁移到 chi。
- 使用 `/api/v1` 路由组统一挂载 API 鉴权。
- 保留 `/healthz` 无鉴权。
- 保持现有 handler 行为和响应格式不变。
- 跑通现有测试，确认无行为回归。

## 阶段二：HTTP 层整理

- 拆出通用中间件：
  - `noSniff`
  - API token auth
  - recover / request id / basic logging
- 整理 JSON 请求和响应工具。
- 统一 path/query 参数解析。
- 保持 handler 代码短小，减少重复校验逻辑。

## 阶段三：服务层边界

- 在 `internal/service` 增加薄服务层。
- 将当前 `httpapi.Store` 大接口拆成更小接口：
  - message
  - todo
  - telemetry
  - weather
  - snapshot
- handler 只负责 HTTP 解析和响应。
- service 负责业务编排和默认值策略。
- store 继续只负责持久化和查询。

## 阶段四：协议增强

- 设计统一错误结构，例如：

```json
{
  "error": {
    "code": "invalid_request",
    "message": "device_id is required"
  }
}
```

- 保留旧字段兼容策略，避免设备端一次性改动过大。
- 增加可选协议能力：
  - 批量消息 ack
  - snapshot `include` 参数
  - telemetry batch 上报
  - 设备信息 header，例如 `X-Device-ID`、`X-Protocol-Version`
- 为消息 ack 和遥测日志设计简单清理策略。

## 阶段五：前端 UI 预留

- 新增 `web/` 目录作为前端项目根目录。
- Go 后端预留静态资源托管能力。
- 生产环境支持 embed 前端 `dist`。
- 开发环境前端 dev server 代理 `/api/v1` 到 Go 后端。
- API 路径保持 `/api/v1/*`，前端页面挂载到 `/`。
